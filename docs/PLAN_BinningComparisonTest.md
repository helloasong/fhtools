# 分箱策略对比测试计划

**目标**: 验证 SmartMonotonicBinner v2.0 相比其他分箱策略的优越性  
**日期**: 2026-03-12  
**测试周期**: 预计 2-3 天

---

## 一、评估维度与标准

### 1.1 核心指标（必须领先）

| 指标 | 权重 | 优秀标准 | 合格标准 |
|-----|------|---------|---------|
| **单调达成率** | 40% | >95% | >85% |
| **100%有解率** | 30% | =100% | >95% |
| **IV损失** | 20% | <5% | <15% |
| **执行速度** | 10% | <2秒 | <5秒 |

### 1.2 测试场景（10种数据模式）

| 场景 | 数据特征 | 难度 | 预期挑战 |
|-----|---------|------|---------|
| 1. 强线性相关 | 完全单调数据 | ⭐ | 验证基础能力 |
| 2. 弱相关波动 | 弱相关+噪音 | ⭐⭐⭐ | 考验抗噪能力 |
| 3. U型关系 | 非单调曲线 | ⭐⭐⭐⭐ | 考验适应性 |
| 4. 长尾分布 | 指数分布 | ⭐⭐ | 考验异常值处理 |
| 5. 双峰分布 | 两个峰值 | ⭐⭐⭐ | 考验多模态处理 |
| 6. 小样本 | n=500 | ⭐⭐⭐ | 考验小样本稳定性 |
| 7. 高基数类别 | 1000+类别 | ⭐⭐⭐⭐ | 考验类别处理 |
| 8. 极端不平衡 | 99:1正负比 | ⭐⭐⭐ | 考验不平衡处理 |
| 9. 完全随机 | 无相关性 | ⭐⭐ | 考验保底机制 |
| 10. 实际业务数据 | mock_data.xlsx | ⭐⭐⭐ | 综合验证 |

## 二、对比策略清单

| 策略 | 类名 | 类型 | 预期表现 |
|-----|------|------|---------|
| 等频分箱 | EqualFrequencyBinner | 无监督 | 高有解率，低单调率 |
| 等宽分箱 | EqualWidthBinner | 无监督 | 高有解率，低单调率 |
| 决策树 | DecisionTreeBinner | 监督 | 中等单调率，可能过拟合 |
| 卡方分箱 | ChiMergeBinner | 监督 | 中等单调率，可能无解 |
| Best-KS | BestKSBinner | 监督 | 趋向单调，但非保证 |
| **智能单调** | **SmartMonotonicBinner** | **监督+优化** | **目标: 全面领先** |

## 三、测试流程

```
Phase 1: 数据准备 (0.5天)
    ├── 生成10种场景数据
    ├── 数据质量验证
    └── 数据文档记录

Phase 2: 测试框架 (0.5天)
    ├── 编写对比测试框架
    ├── 统一评估接口
    └── 结果收集和可视化

Phase 3: 执行测试 (0.5天)
    ├── 每种策略 × 10种场景
    ├── 收集所有指标
    └── 生成对比报告

Phase 4: 分析优化 (1天+)
    ├── 识别薄弱环节
    ├── 针对性优化
    └── 重新测试验证

Phase 5: 文档总结 (0.5天)
    ├── 编写优化记录
    ├── 更新AGENTS.md
    └── 最终报告
```

## 四、优化决策树

```
测试结果分析
    │
    ├── 单调率不达标？
    │   ├── 是 → 强化后处理（切点微调/PAVA）
    │   └── 否 → 继续
    │
    ├── IV损失过大？
    │   ├── 是 → 优化合并策略，减少过度合并
    │   └── 否 → 继续
    │
    ├── 速度慢？
    │   ├── 是 → 优化算法复杂度，减少搜索空间
    │   └── 否 → 继续
    │
    └── 结论：达到最优或权衡接受
```

## 五、成功标准

### 5.1 必须达成（P0）
- [ ] 单调达成率 > 90%（其他策略平均 < 60%）
- [ ] 100%有解率（其他策略存在无解情况）
- [ ] IV损失 < 10%（其他策略部分>20%）

### 5.2 期望达成（P1）
- [ ] 单调达成率 > 95%
- [ ] 平均执行时间 < 1秒
- [ ] 用户满意度评分 > 4.5/5

### 5.3 理想达成（P2）
- [ ] 在所有10种场景下全面领先
- [ ] 零人工干预即可达到专家级效果

## 六、风险控制

| 风险 | 可能性 | 应对策略 |
|-----|-------|---------|
| 某些场景下不如专用算法 | 中 | 明确使用场景，不追求全能 |
| 过度优化导致代码复杂 | 中 | 保持代码清晰，做好注释 |
| 测试数据不代表真实场景 | 中 | 使用真实mock数据验证 |
| 优化陷入局部最优 | 低 | 多轮测试，全局评估 |
